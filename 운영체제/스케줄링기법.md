**1. 선입 선처리 (FCFS, First-Come, First-Served) 스케줄링**

가장 단순한 형태의 스케줄링으로, 준비 큐(Ready Queue)에 도착한 순서대로 프로세스를 디스패치하는 FIFO(First-In, First-Out) 방식입니다. 

*   **특징:** 일단 CPU를 할당받으면 완료될 때까지(혹은 I/O가 발생할 때까지) CPU를 반환하지 않는 **비선점형(Non-preemption) 정책**입니다.

*   **단점 (호위 효과, Convoy Effect):** 실행 시간이 긴 프로세스 하나가 CPU를 차지하고 있으면, 실행 시간이 짧은 여러 프로세스가 그 뒤에서 오래 대기해야 하므로 평균 대기 시간(Average waiting time)이 매우 길어질 수 있습니다.

**2. 최단 작업 우선 (SJF, Shortest-Job-First) 스케줄링**

각 프로세스의 **다음 CPU 버스트 길이를 기준**으로, 가장 짧은 시간을 요구하는 프로세스에게 먼저 CPU를 할당하는 방식입니다.

*   **버스트 길이 예측:** 운영체제는 다음 CPU 버스트의 실제 길이를 미리 알 수 없으므로, 과거의 기록을 바탕으로 **지수 평균(Exponential Averaging)** 기법을 사용하여 예측($\tau_{n+1}$)해야 합니다. 공식 $\tau_{n+1} = \alpha t_n + (1 - \alpha)\tau_n$에서 $\alpha$ 값을 조절하여 최근의 실제 버스트 시간($t_n$)과 과거의 예측치($\tau_n$) 사이의 가중치를 결정합니다.

**3. 우선순위 (Priority) 스케줄링**

각 프로세스에 정수 형태의 우선순위 번호가 부여되며, 가장 높은 우선순위를 가진 프로세스에게 CPU를 할당합니다.

*   **문제점 (우선순위 역전, Priority Inversion):** 비선점형 환경 등에서 우선순위가 낮은 프로세스 때문에 우선순위가 높은 프로세스가 실행되지 못하는 문제가 발생할 수 있습니다.

*   **해결책 (에이징, Aging):** 시간이 지남에 따라 대기 중인 프로세스의 우선순위를 점진적으로 높여주어 기아 상태(Starvation)를 방지합니다.

*   **HRRN (Highest Response Ratio Next):** 대기 시간이 길어질수록 우선순위가 높아지도록 설계된 방식으로, 응답 비율((대기 시간 + 요구 CPU 시간) / 요구 CPU 시간)이 가장 높은 프로세스를 다음으로 선택합니다.

**4. 라운드 로빈 (RR, Round Robin) 스케줄링**

FCFS를 기반으로 하되, **타임 슬라이스(Time slice) 또는 타임 퀀텀(Time quantum)**이라는 제한된 시간 동안만 프로세스가 CPU를 사용할 수 있도록 하는 선점형(Preemptible) 방식입니다.

*   **퀀텀 크기의 중요성:** 퀀텀이 너무 크면 사실상 FCFS와 다를 바 없게 되고, 너무 작으면 프로세스 실행보다 문맥 교환(Context Switch)에 더 많은 시간을 낭비하게 되어 시스템 성능이 저하됩니다. 따라서 대화형 프로세스가 I/O 요청을 발행할 수 있을 정도의 적절한 중간 크기로 설정해야 합니다.

*   **예시 (MS-Windows):** 타이머 인터럽트에 의해 선점되면 Ready 큐의 맨 뒤로 가지만, 더 높은 우선순위 프로세스에 의해 선점될 경우 큐의 맨 앞으로 이동합니다.

**5. 다단계 큐 (Multilevel Queue) 스케줄링**

준비 큐를 성격이 다른 여러 개의 큐(예: 대화형 작업을 위한 Foreground 큐, 배치 작업을 위한 Background 큐)로 분할하여 관리하는 방식입니다.

*   **특징:** 각 큐는 자신만의 스케줄링 알고리즘을 가집니다 (예: Foreground는 RR, Background는 FCFS).

*   **큐 간의 스케줄링:** 큐 사이에도 스케줄링이 필요한데, Foreground를 먼저 처리하는 고정 우선순위 방식을 쓰거나(기아 현상 발생 가능), 전체 CPU 시간의 80%는 Foreground에, 20%는 Background에 할당하는 타임 슬라이스 방식을 적용할 수 있습니다.

**6. 다단계 피드백 큐 (Multilevel Feedback Queue) 스케줄링**

다단계 큐의 한계를 극복하기 위해, **프로세스가 여러 큐 사이를 이동(Move)할 수 있도록 허용**하는 방식입니다.

*   **특징:** 대기 시간이 길어지면 상위 큐로 이동시키는 에이징(Aging) 기법을 자연스럽게 구현할 수 있습니다. 큐의 개수, 각 큐의 스케줄링 알고리즘, 프로세스를 상위/하위 큐로 승격/강등시키는 조건 등에 의해 시스템이 유연하게 정의되며, UNIX/Linux의 비실시간 태스크 스케줄링의 기본 골격으로 사용됩니다.

**7. 공정 분배 (FSS, Fair Share Scheduling)**

개별 프로세스 단위가 아니라, **사용자(또는 그룹) 단위로 시스템 자원의 접근을 제어**하는 방식입니다.

*   **특징:** 덜 중요한 그룹이 자원을 독점하는 것을 방지하며, 각 그룹에 사전에 할당된 자원 비율(%)에 따라 스케줄링이 이루어집니다. 그룹에 부여된 크레딧(Credit)에 따라 하위 그룹의 우선순위가 높은 프로세스보다 상위 그룹의 우선순위가 낮은 프로세스가 먼저 실행될 수도 있습니다.

**8. 실시간 (Real-Time) 및 데드라인 기반 스케줄링**

*   **실시간 스케줄링:** 우선순위 역전을 방지하고 빠른 반응을 보장하기 위해 선점형 커널(Preemptive kernel)이 필수적입니다.

*   **데드라인 기반 스케줄링:**

    *   **RMS (Rate Monotonic Scheduling):** 프로세스의 주기(Cycle)가 짧을수록 더 높은 정적 우선순위를 부여하는 방식입니다.

    *   **EDF (Earliest Deadline First):** 마감 시간(Deadline)이 가장 임박한 태스크에 가장 높은 동적 우선순위를 부여합니다.

**9. 다중 프로세서 (Multiple-Processor) 스케줄링**

*   **부하 분산 (Load Balancing):** 특정 프로세서에 부하가 집중되지 않도록 Push 또는 Pull migration을 통해 작업을 분산시킵니다.

*   **프로세서 친화성 (Processor Affinity):** 캐시 적중률(Cache hit)과 메모리 접근 속도(NUMA 구조 등)를 위해 프로세스가 동일한 CPU에서 계속 실행되도록 유지하려는 성질입니다. 이를 보장하지 않는 연성 친화성(Soft affinity)과, 이동을 엄격히 금지하는 경성 친화성(Hard affinity)으로 나뉩니다.
