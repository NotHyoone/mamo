# commit rule

## 1. 커밋 메시지 작성 규칙

- 커밋 메시지는 명확하고 간결하게 작성해야 합니다.
- 커밋 메시지는 현재 시제로 작성해야 합니다.

### 1.1 커밋 메시지 형식

```<타입>(<범위>): <제목>
<본문>
```

- 타입: feat, fix, docs, style, refactor, test, chore 등
- 범위: 변경된 기능이나 파일의 범위를 명시 (선택 사항)
- 제목: 변경 사항을 간략하게 설명
- 본문: 변경 사항에 대한 자세한 설명 (선택 사항)

### 1.2 예시

```bash
feat(auth): 로그인 기능 추가
fix(api): API 응답 오류 수정
docs: README 파일 업데이트
style: 코드 포맷팅 적용, 세미콜론 누락, 코드 변경이 없는 경우
refactor: 코드 리팩토링
test: 테스트 코드, 리팩토링 테스트 코드 추가
chore: 빌드 업무 수정, 패키지 매니저 수정, 코드 변경이 없는 경우
```

## 2. 커밋 메시지 작성 시 주의사항

- 커밋 메시지는 변경 사항을 명확하게 설명해야 합니다.
- 커밋 메시지는 팀원들이 이해할 수 있도록 작성해야 합니다.
- 커밋 메시지는 변경 사항의 이유와 방법을 포함해야 합니다.
- 커밋 메시지는 변경 사항이 어떤 문제를 해결하는지 설명해야 합니다.
- 커밋 메시지는 변경 사항이 어떤 기능을 추가하는지 설명해야 합니다.
- 커밋 메시지는 변경 사항이 어떤 버그를 수정하는지 설명해야 합니다.
